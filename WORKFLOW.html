The Agent Workflow

Start here

A planner → builder → reviewer → integrator agent pipeline for Claude Code, built on git worktrees: several builder agents work in parallel with zero merge collisions, and you — the human — approve exactly two things per feature: the plan, and the merge.

Who this is for: anyone adopting the framework in their own project, and any agent that needs to understand its place in the pipeline.

  • Two human gates only — plan approval and merge approval. Everything between them (building, review, verification, status upkeep) is automated.
  • Isolation is the trick — every feature is built entirely inside its own folder; shared-file edits are deferred, not made, so parallel branches can never conflict.
  • The chain is self-propelling — each agent writes the next agent's instructions into HANDOFF.md, so launching the next one is a one-line paste.
  • You never need a terminal — agents run all commands themselves; you only chat and approve.

The big picture

The framework turns one repo into a small fleet. The main checkout runs the orchestrator (your project console); each in-flight feature gets its own git worktree on a feat/<name> branch, where a builder works. A reviewer is dispatched into a feature's worktree when it's done, and a single serialized integrator merges verified features to main.

Every arrow in that diagram is a → Next: block the finishing agent writes into the feature's HANDOFF.md — see the handoff contract. You approve the merge from the knowledge base's Verification page (screenshot evidence per acceptance criterion), not by hand-testing.

No model or thinking pins. The framework configures nothing the Claude Code app already handles: role files carry no model: frontmatter and kickoff cards carry no thinking directives. You pick model and thinking depth per session in the app — heavier for planning and orchestration, lighter for mechanical runs — so credit spend stays in your hands.

The roles

Each role is defined once, in ~/.claude/agents/<role>.md. That single file is both the spawnable subagent (the Agent tool can dispatch it) and the role's job description (the /role command loads it into a long-lived session).

RoleHow manyLives inMay editJob
orchestrator Exactly one main worktree (main branch) Program PRD, FLEET.md — never feature code Project lead / console. Holds whole-project scope, briefs planners, maintains the dashboard, sequences and recommends work. Starts nothing on its own — it observes, reports, and acts only when told.
planner On demand, per feature dispatched (or the feature worktree) Docs only — never src/ Grills you one question at a time, then writes the Feature PRD (docs/features/<NAME>.md) and cuts it into vertical-slice issues with detailed plans. Stops at the plan gate.
builder Many — one per worktree (your parallelism) its feature worktree (feat/<name>) Only its own src/features/<name>/ folder (+ its own feature PRD) Implements the PRD's issues in blocked-by order, test-first at the PRD's pre-agreed seams (red → green). Commits every checkpoint; stamps status; never touches shared files.
reviewer On demand, dispatched per feature the feature's own worktree Only REVIEW.md, VERIFY/, PRD evidence links, HANDOFF.md, .status.json — never app code Three phases in one dispatch: review the diff (reads), verify it works (build, tests, dev harness), and capture the screenshot evidence pack the owner approves from. Committed on the feature branch so it merges with the feature.
integrator Exactly one, strictly serialized main worktree (main branch) Danger Zone — the ONLY role allowed to edit shared files Merges a verified feature to main, applies the deferred INTEGRATION.md edits (routes, nav, types, DB), wires the real adapters, folds the changelog, verifies the whole app, refreshes the evidence pack against the integrated app.

Review vs. verify are phases, not roles. The review phase answers "is this code right?" by reading; the verify phase answers "does it actually work?" by running. Code routinely passes one and fails the other, so the reviewer always does both, in order, then turns the result into screenshots.

Automatic role adoption

Roles are assigned by branch, via a rule in ~/.claude/CLAUDE.md: opening a Claude Code session in a worktree checked out on main auto-adopts the orchestrator; on feat/* it auto-adopts the builder for that folder. There is no dedicated review worktree — to review a feature, open its own worktree and run /role reviewer. Override any auto-role with /role <name>.

Skills & commands

Six skills (slash commands, installed at ~/.claude/commands/) drive the framework. Most of the time you don't need them — the auto-role rule plus a pasted kickoff card does the same work — but they're the explicit interface.

CommandWhat it doesTypically run by
/fleet Adopts the orchestrator and runs a fleet cycle: regenerates the FLEET.md dashboard from every worktree's status, checks alignment to the Program PRD, and reports project state — what's in flight, what awaits your gate, what to start next. Recommends only; starts nothing. You, in the main worktree — your "what's going on?" button.
/grill [topic] A relentless one-question-at-a-time interview (each question comes with a recommended answer) until a plan's decision tree has no unresolved branches. The orchestrator and planner apply this behavior automatically before writing any PRD. You, anytime you want a design pressure-tested; built into orchestrator + planner.
/bootstrap <idea> Stands the whole framework up in a brand-new repo: grills you about the idea, writes the Program PRD with a thin milestone 1, creates the feature-isolation convention + FLEET.md roster, stands up the HTML knowledge base, sets up git (asking once about an online repo), creates the first feat/* worktrees, and hands them to the planner. Deliberately starts with 1–2 features. You, once, in a fresh project folder.
/role <name> Adopts a role (orchestrator / planner / builder / reviewer / integrator) for the whole session, overriding the branch-based auto-role. This is how a feature gets reviewed: open its worktree, run /role reviewer. You, when the auto-role isn't the role you need.
/handoff [recipient|same] The end-of-turn ritual for every role: brings the durable records current (PRD progress, CHANGELOG, INTEGRATION.md, .status.json), rewrites the feature's HANDOFF.md (who left it → who picks up, next goals, known problems, doc pointers — never copied content), and prints the next agent's copy/paste kickoff card. Also works mid-task to compact a long context window — hand off to a fresh session of the same role, losing nothing the documents don't hold. Every agent, at the end of every completed turn.
/architecture-review On demand only: scans the whole repo for architecture-deepening opportunities and writes a suggested-improvements page into the knowledge base. Recommendations only — never changes code, never runs automatically. You or the orchestrator, occasionally.

How each agent uses the skills

AgentSkills it runsHow it uses them
orchestrator /fleet · /grill · /bootstrap · /architecture-review · /handoff /fleet is the orchestrator's main loop — dashboard refresh + state-of-the-project report. It applies /grill behavior before any Program-PRD writing, runs /bootstrap for greenfield projects, and runs /architecture-review only when you ask.
planner /grill (built in) · /handoff Grilling is step 1 of planning — the PRD isn't written until every design branch is resolved. When the slice list is approved, /handoff writes the → Next: builder block and prints the builder's kickoff card.
builder /handoff Builders mostly use files, not commands: the PRD is the plan, .status.json is the heartbeat. At the end of each completed turn (or to compact a long session), /handoff updates the records and prints the reviewer's (or next builder session's) card.
reviewer /role reviewer (to adopt) · /handoff Adopted inside the feature's own worktree. Its evidence phase uses the zero-dep capture script (scripts/verify-shots.mjs) rather than a skill, then rebuilds the KB so the Verification page is current. /handoff routes pass → integrator, fail → builder.
integrator /fleet (after merge) · /handoff After a successful merge it sets the feature's status to merged, runs /fleet to refresh the dashboard, and hands back to the orchestrator with a suggestion for what the freed worktree should pick up next.
The human dispatches every agent. No agent starts another agent. When an agent's work is done, it ends with a kickoff card; you paste that card into the next session (or tell the orchestrator to dispatch it). If an agent needs a decision instead, it asks and waits — no card until the blocker is resolved.

A feature's lifecycle, step by step

  1. Greenlight. You add the feature to the Program PRD's features list (or ask the orchestrator to; it grills you first). Nothing is built that isn't on this list.
  2. Plan. The planner grills you, writes the Feature PRD — problem · solution · user stories · implementation decisions · testing seams · out-of-scope — and cuts it into vertical-slice issues, each with a detailed plan, acceptance criteria, tests-first notes, blocked-by ordering, and an Evidence: pending review line. GATE 1 You approve the numbered slice list.
  3. Build. A worktree is created (git worktree add ../<dir> -b feat/<name>) and the builder works the issues in order, test-first at the PRD's seams, committing each checkpoint and stamping .status.json. It never waits between checkpoints — it self-continues until done or genuinely blocked.
  4. Review + verify + evidence. Status hits completed — pending review; the reviewer is dispatched into that worktree. Phase 1 reads the diff (scope discipline, three-layer purity, plan fidelity, correctness). Phase 2 runs it (typecheck, in-folder tests, dev harness). Phase 3 captures one screenshot per acceptance criterion into VERIFY/ and rebuilds the KB. Verdicts go into REVIEW.md; pass → verified — pending integration, fail → changes requested back to the builder.
  5. Approve from evidence. GATE 2 You skim the feature's evidence pack on the KB's Verification page and approve the merge — no hand-testing.
  6. Integrate. The integrator (one at a time) merges the branch, applies every deferred INTEGRATION.md edit, wires real adapters + routes/nav, folds the feature changelog into the app changelog, marks the PRD entry done, builds and tests the whole app, and re-captures the evidence pack against the integrated app ("passed alone, broke wired in" is the classic failure).
  7. Recycle. Status merged; /fleet refreshes the dashboard; the worktree is free for the next feature.

Status vocabulary (used in .status.json and the <!--fleet--> header): planning · in progress · completed — pending review · in review · changes requested · verified — pending integration · integrating · merged · blocked · idle.

The knowledge base (KB) — the human view

Authored markdown under docs/ is built into a cross-linked HTML site under docs-site/ by a zero-dependency Node script (npm run build:docs). Three pages are generated live from the fleet's files: Status (the dashboard), Kickoff (launch cards with a generator), and Verification (every feature's VERIFY/ evidence pack, aggregated from every worktree — merged or not). House rule: after editing any authored doc, rebuild — never leave the HTML stale, and never hand-edit generated output.

The handoff contract — the self-propelling chain

Every role orients on start and hands off on finish, so launching the next agent takes one line and it learns the rest from the files.

  • Orient: your instructions are the → Next: <your role> block in the feature's HANDOFF.md, read together with the Feature PRD and the Program PRD. Confirm the feature, then proceed.
  • Hand off: bring the durable records current first (the handoff is never the only home of information), then rewrite HANDOFF.md and end your reply with the next agent's kickoff card.

      <Components>.tsx        #   thin React views
      <name>.css              #   ALL classes prefixed with a unique namespace
      __dev__/harness.html    #   standalone preview with mock data
      __tests__/              #   in-folder tests (registered at integration)
      CHANGELOG.md · INTEGRATION.md · HANDOFF.md · .status.json
      VERIFY/                 #   evidence pack — written by the REVIEWER only

The three-layer split (why every folder looks like that)

  1. logic.ts — pure, portable core. Plain TypeScript: types, reducers, math, scoring. No React, no DOM, no platform APIs, no app imports. Ports verbatim to any platform.
  2. View — thin React components. Render state, dispatch actions. A future mobile port re-implements only this layer.
  3. Adapters — the injected platform seam. A small interface for anything platform-specific (persistence, networking, images), received as a prop, shipping an in-memory default so the harness runs standalone. The integrator supplies the real implementation at merge.

The Danger Zone integrator only

The shared files that are edit-magnets — an edit here is how parallel work collides. Feature agents never touch them; every needed change is written into INTEGRATION.md instead and applied during the merge pass. Typical list (each project keeps its own in docs/features/README.md): the app shell and router, shared types, the DB layer, shared UI components and styles, the entry point, build config (package.json, vite.config, tsconfig), the shared test runner, everything in docs/ except your own feature PRD, and other features' folders.

Git & GitHub rules

Branch model

  • One repo, many worktrees. main is checked out in the primary folder; each feature gets a sibling worktree via git worktree add ../<dir> -b feat/<name> — parallel branches side by side, no stash-and-switch.
  • One branch per feature (feat/<name>), off main, working only inside its feature folder. Because branches only add their own folder, merges stay clean in any order.
  • Keep branches short-lived — the less a branch drifts from main, the smaller any conflict.

Commit rules (builders)

  • Commit at every checkpoint on the feature branch — small, reviewable steps — with conventional messages: type(scope): summary (e.g. feat(storage3d): open box on tile click).
  • Stage only your own folder (git add src/features/<name>/) — never git add -A. This keeps other features' in-progress files and another session's edits out of your commits.
  • Gate every checkpoint: typecheck + tests pass before moving on; a failing gate is fixed before continuing.
  • Docs travel with code: PRD progress notes and the feature's own CHANGELOG.md ride in the same commit as the change that caused them.

Hard lines human-gated

  • No agent ever commits on main, merges, tags, or pushes on its own. The owner approves merges (Gate 2); the integrator performs them, one feature at a time — never two integrations at once.
  • Pushing is human-gated, like merging. After an approved merge, the integrator offers to push main — it pushes only on your OK, and never force-pushes. Feature branches are pushed only when you ask (to back up work or open a PR).
  • The reviewer commits only its own artifacts (REVIEW.md, VERIFY/, status files) on the feature branch — never app code.

Remote setup (asked once, at bootstrap)

During /bootstrap the orchestrator initializes git, writes a .gitignore (node_modules, build output, secrets/env, local DB files, large data packs), makes the initial commit, and asks one question: "Will this project use an online repo (GitHub/GitLab/…)? If so, what's the URL — and should I create it?" Yes → wire the remote and push main, recording in FLEET.md that a remote exists. No → everything stays local; the worktree pipeline needs no remote. The question is never re-asked per cycle.

General house rules — every role inherits these

  • Build entirely inside your feature folder. Touch nothing in the Danger Zone; defer every shared edit to INTEGRATION.md, precise enough to apply blind. "If possible" means "isolate everything you can and defer the rest" — never "edit shared files because it was convenient."
  • Stay buildable in isolation. Every feature folder must build, run via its own __dev__ harness, and pass its in-folder tests against mock data, without the rest of the app wired up. (Vite serves any HTML by URL, so the harness needs no config edits.)
  • Test-first at the PRD's seams. The planner pre-agrees the public boundaries where behavior is verified; builders write the failing test at the seam first (red), implement the minimum to pass (green), and note deeper refactors for the reviewer rather than doing them mid-loop. If an issue needs a seam the PRD doesn't name, that's a plan change — flag it, don't invent one.
  • Scope discipline. Anything new goes on the Program PRD's features list before it's built. No scope expansion mid-task.
  • Namespace all CSS with a unique per-feature prefix so two features can never clash. Reading shared design tokens is fine; adding to shared styles is not.
  • Self-continue — don't go idle. Finish a checkpoint → commit + stamp status → immediately start the next one. The only legitimate stop is a genuine decision: set state to blocked/awaiting-approval with the question filled in — never stop silently.
  • Status is pull-based. Agents stamp their own .status.json on every transition; /fleet aggregates when you ask. No polling loops, no always-on watcher.
  • Evidence over claims. The reviewer quotes real output (test summaries, error text) and looks at every screenshot it captures — a blank PNG is a failed capture, not evidence. You approve merges from the Verification page, not from assertions.
  • The KB is baseline, not optional. Every framework project ships the HTML knowledge base; after editing an authored doc, rebuild it. Never hand-edit anything generated.
  • Escalation goes through the chain. The reviewer routes fixes back to the builder via the handoff; only scope/plan changes and serious flaws (isolation violated, design-level or security problems) surface to the human via the orchestrator.

Launching agents — kickoff cards

Each role launches from a short paste-able card (the full set lives in ~/.claude/KICKOFF.md; projects with the KB also get a Kickoff page with an interactive generator). Open a session in the right worktree — the auto-role rule already applies the role from the branch — paste the card, and the agent reads its role file + HANDOFF.md + the feature PRD and takes it from there. Since every finishing agent prints the next agent's card, day-to-day operation is:

Paste the card the last agent printed → approve at the two gates → repeat.

Example (builder):

You are a Builder Agent assigned to the worktree {{WORKTREE}}. Read
~/.claude/agents/builder.md and operate under it. Orient: read the Program PRD
(docs/PRD.md), your feature's PRD in docs/features/, and this worktree's
HANDOFF.md — the "→ Next: builder" block is your instructions. Confirm the
feature, then build it inside your src/features/<name>/ folder only, working
the PRD's issues in blocked-by order, test-first at the PRD's seams. Check each
issue off as it closes. Update HANDOFF.md when you pause or finish.

Installing / porting to another machine

The framework is a folder of markdown + three zero-dependency Node scripts. To adopt it:

  1. Prerequisites: Claude Code (app or CLI) · git · Node.js (any recent LTS) · Edge or Chrome (only for screenshot evidence packs on UI projects). No npm packages are installed.
  2. Copy the kit (this folder) to the new machine, then copy into ~/.claude/: agents/ (the five roles), commands/ (the six skills), AGENTS_FRAMEWORK.md, KICKOFF.md, and add the auto-role block to ~/.claude/CLAUDE.md. Nothing in it is secret.
  3. New project: open Claude Code in an empty folder and run /bootstrap <your idea> — it stands everything up, including the knowledge base, from the kit's kb/ starter.
  4. Existing project: the roles degrade gracefully — the planner simply creates the docs/features/ convention on first use.

Deeper reference: AGENTS_FRAMEWORK.md (the single source of truth) · KICKOFF.md (all launch cards) · agents/*.md (full role definitions) · each project's docs/features/README.md (its exact Danger Zone list and folder convention) and its knowledge base's Framework, Status, Kickoff, and Verification pages.
